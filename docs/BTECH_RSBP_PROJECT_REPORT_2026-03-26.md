## Abstract
Reverse Shell Behavior Profiler (RSBP) is a Linux security daemon that detects reverse-shell tradecraft by correlating low-level system call telemetry with userspace behavioral analytics. The system uses an eBPF program attached to Linux syscall tracepoints to observe process execution and network behavior with low overhead. RSBP emits compact event records for key transitions in reverse shell activity, including process execution (execve), network socket creation (socket), outbound connections (connect), stream redirection used to bind an interactive shell to a network channel (dup2 and dup3), and process plumbing indicative of staged command execution (fork, clone3, pipe, and pipe2). These events are streamed to userspace through a kernel ring buffer, time-normalized, and aggregated into short-lived per-process sessions.

In userspace, a correlation engine maintains a process-tree aware session model that merges evidence across parent–child relationships, enabling reliable detection even when reverse-shell tooling divides responsibilities across multiple processes. A detection engine then computes confidence scores from both heuristic chain features and explicit rules that capture common reverse shell patterns such as /dev/tcp-based shells, Python socket shells, socat relays, and netcat-style callbacks. To reduce false positives without losing recall for high-risk behaviors, RSBP applies whitelist and suppression logic only when sessions are incomplete or low-confidence, and it supports configurable thresholds and private-network testing modes. Post-detection stages enrich alerts with host, process, and network context and optionally collect lightweight forensic artifacts from /proc.

The project’s academic value lies in demonstrating an end-to-end, behavior-based detection pipeline that combines modern kernel instrumentation (eBPF), formal event correlation, rule-based scoring, and measurable operational monitoring. RSBP serves as a practical case study in systems security engineering, showing how telemetry fidelity, correlation design, and observability jointly determine detection reliability.

## System Architecture
RSBP follows a layered architecture in which kernel-space instrumentation provides trustworthy, fine-grained telemetry, while userspace components implement correlation, detection, enrichment, forensics, and delivery. At the lowest layer, eBPF tracepoint programs attach to Linux syscall entry and exit points for execve and socket, and syscall entry points for connect, dup2, dup3, fork, clone3, pipe, and pipe2. This syscall set is sufficient to model the defining elements of reverse shells: the creation or reuse of a network channel, an outbound connection to a remote controller, and redirection of standard I/O so that a shell becomes interactive over that channel. By restricting kernel work to compact event capture and by delegating policy logic to userspace, RSBP keeps kernel overhead small and enables rapid iteration of detection logic.

The userspace pipeline begins with an event collector that reads ring-buffer records and converts them into structured syscall events enriched with arrival timestamps. A correlation engine then aggregates events into a session state keyed by process identifier and augmented by a process tree. The session state contains boolean evidence flags, network endpoint fields, and a temporal window that bounds how long a session may evolve before expiring. Correlation also propagates connect evidence from child processes to their parents where appropriate, which is essential in real reverse-shell executions where a wrapper or interpreter spawns a child that performs the connect or I/O redirection. When a session satisfies completion criteria, it is emitted as a completed behavioral unit for scoring.

The detection stage evaluates completed sessions using a confidence scoring function and a set of weighted rules. The scoring function emphasizes behavior chains consistent with reverse shells, such as execve+connect with dup-to-stdio, or connect behavior by known interactive tools even when not all flags are observed within a short window. In parallel, weighted rules capture additional indicators such as suspicious command traits, known command-and-control port usage, and external remote endpoints. Suppression mechanisms reduce noise by ignoring benign system and developer-tool activity unless a session exhibits the defining reverse-shell behavior. After detection, an enrichment stage adds network intelligence (GeoIP, ASN, reputation), reverse DNS, and process-level context gathered from /proc. Finally, a forensics stage can collect lightweight artifacts such as file-descriptor listings, memory map excerpts, socket tables, and an optional mini capture artifact for evidence packaging.

Alert delivery is implemented as a multi-sink output router. RSBP can persist alerts locally as newline-delimited JSON (JSONL) and can forward alerts to downstream systems including Elasticsearch for indexing, Splunk via HTTP event collector style payloads, Kafka for stream processing, Syslog in RFC5424-like format for SIEM ingestion, and generic webhooks with HMAC signatures. A degraded-mode sink design allows a primary sink such as Elasticsearch to fail without blocking alert emission, routing alerts to a JSONL fallback while buffering a bounded number of messages for replay once the primary recovers.

Operational monitoring is provided via Prometheus metrics exposed by the daemon and a containerized monitoring stack. Prometheus scrapes daemon metrics and Grafana provides dashboards for system throughput, pipeline health, and sink emission performance. Filebeat ingests JSONL alerts from the host into Elasticsearch indices for search and visualization. Kibana provides interactive investigation and ad-hoc queries over indexed alerts. The monitoring stack is orchestrated using Docker Compose, with services for Prometheus, Elasticsearch, Kibana, Grafana, and Filebeat connected over an internal bridge network.

## Internal Modules
The internal/ebpf module implements kernel telemetry collection for continuous monitoring. It loads compiled eBPF programs and maps, attaches them to selected syscall tracepoints, and streams events through a ring buffer into userspace. The loader converts kernel time to wall-clock time using boot-time inference so downstream correlation uses consistent timestamps and stable ordering. Events include process identity, executable metadata, credentials, and network endpoint fields in a compact structure designed for high-throughput delivery. The module validates prerequisites for safe operation, including required capabilities, kernel feature availability, and successful attachment of all expected probes. It exports Prometheus metrics for event counts by syscall, ring-buffer drops, and attachment status, enabling operators to tune buffer sizing and polling to minimize loss. It can periodically verify probe state and reattach when drift is detected, improving resilience across restarts and long uptimes. Overall, kernel work is limited to capture, while policy, validation, and recovery remain in userspace.

The internal/correlation module transforms syscall streams into short-lived behavioral sessions keyed by PID. It updates evidence flags and network fields as events arrive and tracks parent–child relationships so sessions reflect execution lineage and tooling handoffs. Because reverse shells often split actions across processes, the engine can merge and propagate connect evidence across related processes so a session retains a concrete remote endpoint even when the initiating shell did not call connect directly. Completion is time-bounded and behavior-centric, requiring a credible remote connection while filtering trivial local traffic and known-safe endpoints, with an optional private-network mode for lab testing and demonstrations. The module labels sessions against a small set of templates such as /dev/tcp, Python socket, socat, and netcat-style callbacks to support explainability. It also deduplicates emissions over a bounded interval so repeated retries or looping callbacks do not produce uncontrolled alert storms.

The internal/detection module assigns confidence and severity to completed sessions using a heuristic score combined with weighted rules. The base score reflects evidence flags, endpoint properties, and tool-aware heuristics, while rules add indicators such as suspicious command traits, characteristic syscall chains, external addressing, and unusual or command-and-control-like port behavior. Rule weights and thresholds are configurable so environments with high developer activity can reduce noise without suppressing true interactive callbacks. Suppression and whitelisting reduce false positives using process identity, executable path, user context, and IP ranges, but suppression is relaxed when the behavioral chain is complete or closely matches a reverse-shell template. The module produces structured alerts with scoring context, correlation metadata, fired rule identifiers, and a MITRE ATT&CK technique mapping derived from the rule set to support standardized reporting. It exports metrics for detections, suppressions, and score distributions to support iterative tuning.

The internal/enrichment module adds context that improves triage without altering the detection decision. It enriches remote endpoints with GeoIP location and autonomous system data and can optionally consult reputation sources when an API key is configured for external intelligence. To keep latency bounded, enrichment uses concurrency with strict timeouts, and a TTL cache keyed by remote IP avoids repeated lookups during bursts or repeated callbacks. Failed or slow enrichments degrade gracefully by returning partial context rather than delaying alert emission, which preserves operational timeliness. The module also captures process-centric context from /proc, linking syscall behavior to executable paths, parent process information, open file descriptors, and a snapshot of active connections at alert time. Reverse DNS is applied with an aggressive timeout to provide a readable hostname when available. These fields are written into the alert so downstream systems can group, search, and investigate detections efficiently.

The internal/forensics module collects lightweight evidence for higher-confidence alerts while controlling host impact. It creates a per-alert bundle directory and captures /proc-based artifacts, including a process snapshot, open file descriptor targets, a bounded memory map excerpt, and relevant socket tables. When enabled and permitted by disk policy, it can also write a small capture artifact suitable for evidence packaging. Before collection, the module checks disk space and enforces size limits so that evidence capture cannot exhaust storage on constrained hosts. A manifest records collection time, process liveness, and cryptographic hashes of artifacts to support integrity verification and later audit. Collection runs asynchronously so alert emission is not blocked, and cleanup routines remove aged bundles to keep storage bounded. Bundles use predictable naming to support incident timelines and review. Optional permission hardening and redaction can reduce exposure of sensitive host details, yielding practical forensic readiness while minimizing overhead.

The internal/output module delivers alerts to multiple sinks with resilience under partial dependency failure. It routes alerts to destinations such as JSONL persistence, Elasticsearch indexing, Splunk over HTTPS, Kafka publication, Syslog emission, and generic webhooks with configurable headers and HMAC signing. Delivery is isolated per sink using concurrent workers with per-sink timeouts and metrics for success and failure, which makes sink health observable and simplifies troubleshooting. For primary sinks like Elasticsearch, a degraded-mode strategy buffers a bounded number of alerts and immediately writes to a JSONL fallback so evidence is retained locally even during outages. Recovery attempts replay buffered alerts when the primary becomes available again, and bounded buffering prevents memory growth under sustained failure. JSONL writing uses conservative semantics and rotation to limit corruption risk and unbounded growth, making the local store suitable as an evidence trail and as a backfill source for later indexing.

The internal/api module exposes an operator-facing HTTP interface for health, status, and limited control operations. It provides basic health and uptime endpoints, deeper health views with processed event counts, and a stats endpoint aggregating ingestion, correlation, detection, and output signals in a form suitable for dashboards and automated checks. For investigation, the API can return a recent window of locally persisted alerts and fetch a specific alert by identifier, enabling quick validation even when Elasticsearch or other sinks are unavailable. It supports runtime whitelist management through endpoints that create, read, update, and delete entries, and it provides a reload endpoint for refreshing thresholds, whitelists, and routing without restarting the daemon. A test endpoint injects a synthetic event to verify end-to-end behavior in controlled environments. Optional bearer-token authentication limits access, and a metrics redirect supports operational convenience, with an intentionally minimal surface to reduce operational risk and attack exposure.

## MITRE ATT&CK Techniques Detected
RSBP associates its detections with a set of MITRE ATT&CK technique identifiers derived from the rules that fire during evaluation, and it also applies baseline technique identifiers for reverse-shell-like behavior when rule-specific mapping is absent. In the current implementation, the technique coverage includes T1059 (Command and Scripting Interpreter) to represent interpreter and shell execution as the core execution mechanism of reverse shells; T1104 (Multi-Stage Channels) to represent the multi-process and staged-channel nature of reverse shells and their control flows; and T1571 (Non-Standard Port) to represent command-and-control over ports that are atypical for legitimate application traffic. The mapping further includes T1071 (Application Layer Protocol) when enrichment or intelligence-oriented indicators are present, T1048 (Exfiltration Over Alternative Protocol) for high-numbered and atypical port behaviors used as a proxy for unconventional data movement channels, and T1029 (Scheduled Transfer) to represent time-window anomalies that may indicate automated or off-hours activity patterns. These identifiers are carried into alert metadata to support downstream reporting, threat-hunting queries, and academic evaluation against a standardized adversary technique framework.

## Design Decisions: Why eBPF over auditd and ptrace
RSBP adopts eBPF tracepoint instrumentation because the design objective is to capture high-fidelity behavioral evidence at the point where operating system semantics are enforced, while keeping overhead and operational complexity low. Traditional auditd pipelines rely on kernel audit logs that can become extremely noisy for syscall-level monitoring, leading to large storage costs, increased parsing overhead, and delayed visibility under bursty workloads. In contrast, an eBPF program can be engineered to emit only the minimal fields needed for reverse-shell correlation and scoring, substantially reducing data volume and enabling near real-time userspace processing.

Ptrace-based monitoring, while powerful for debugging, is poorly suited for always-on behavioral security monitoring at host scale. It imposes significant per-process overhead, requires intrusive attachment semantics, and can interfere with process execution and timing. Moreover, ptrace approaches typically lack the same low-latency, event-stream architecture offered by eBPF ring buffers and are less amenable to clean isolation between telemetry capture and policy logic.

By selecting eBPF, RSBP achieves a principled separation of concerns: the kernel layer performs compact, safe event capture; userspace implements correlation, detection, and integrations where iteration is faster and safer; and the monitoring stack provides measurable observability. This design aligns with modern academic and industrial trends in host-based telemetry, and it enables reproducible experiments where changes to detection logic can be evaluated without altering kernel-space components.

## Limitations and Future Work
RSBP’s design is constrained by practical limitations inherent to eBPF deployment and to short-window correlation. The eBPF loader requires a compatible Linux kernel, BTF availability, and elevated capabilities, which can limit portability across older kernels or restricted environments. The current correlation model is tuned for short behavioral windows and can miss multi-minute staged activities or low-and-slow interactive sessions unless thresholds and windows are carefully adapted. Conversely, tool-aware heuristics and whitelist logic must be tuned to reduce false positives in developer-heavy environments where shells, interpreters, and network connections occur frequently.

The current forensic capture emphasizes low-impact artifacts from /proc and a small evidence bundle; in constrained environments, capture may be disabled or partial. Future work should extend artifact collection with configurable, privilege-aware packet capture using appropriate capture interfaces when permitted, and should enrich evidence with file lineage, TTY associations, and environment-variable risk indicators. The technique mapping can be expanded and made more consistent by aligning rule identifiers with mapping tables and by incorporating additional ATT&CK techniques relevant to reverse shells, such as more detailed command-and-control sub-techniques and defense evasion indicators.

Further improvements include expanding syscall coverage to capture richer network semantics, adding adaptive baselining per host and per user, introducing configurable rule packs for different environments, and implementing offline evaluation tooling that measures precision, recall, and latency on labeled workloads. Finally, a packaging-oriented deployment model with hardened defaults, signed rule sets, and automated dashboard provisioning would strengthen the project’s readiness for real-world use and would support more rigorous academic evaluation.

## Go Dependencies and Their Purpose
RSBP’s Go module dependencies reflect its end-to-end scope, spanning kernel telemetry, HTTP serving, metrics, storage integrations, and enrichment. The dependency github.com/cilium/ebpf provides the core eBPF loading, linking, and ring-buffer reading primitives used to attach tracepoint programs and stream events to userspace. The dependency github.com/elastic/go-elasticsearch/v8 supplies the official Elasticsearch client used for bulk indexing of alerts and for index template and lifecycle policy provisioning. The dependency github.com/fsnotify/fsnotify supports alert-tail functionality by watching JSONL files for changes in follow mode. The dependency github.com/go-chi/chi/v5 provides the HTTP router used by the operator API to expose health, stats, and control endpoints.

Configuration parsing and mapping are supported by github.com/go-viper/mapstructure/v2 and gopkg.in/yaml.v3, enabling structured decoding of YAML configuration into Go structs. The dependency github.com/prometheus/client_golang provides metrics instruments and the HTTP handler used to expose Prometheus-compatible metrics. Command-line interface structure is implemented using github.com/spf13/cobra, while structured logging is implemented using go.uber.org/zap.

For enrichment and forensics, github.com/oschwald/geoip2-golang and github.com/oschwald/maxminddb-golang enable local GeoIP lookups, and github.com/jellydator/ttlcache/v3 provides a TTL cache for repeated enrichment results. The dependency github.com/google/gopacket and its supporting layers enable generation of packet-capture artifacts and related network evidence packaging in the forensics module. For stream output, github.com/segmentio/kafka-go provides Kafka client functionality, including optional TLS and SASL mechanisms. The dependency golang.org/x/sync supports structured concurrency patterns used in enrichment, while golang.org/x/sys and golang.org/x/net provide low-level and networking utilities used across the system.

Several indirect dependencies provide performance primitives, compression, protobuf support, and OpenTelemetry wiring used by upstream libraries; these are included as transitive requirements to ensure compatibility and stable builds.

Beyond the primary dependencies, the build includes indirect modules that support telemetry, transport, encoding, and correctness. The Prometheus client stack pulls in github.com/beorn7/perks, github.com/cespare/xxhash/v2, github.com/munnerz/goautoneg, github.com/prometheus/client_model, github.com/prometheus/common, and github.com/prometheus/procfs, which collectively provide efficient counters and histograms, hashing utilities, content negotiation helpers, metric model definitions, shared exposition utilities, and Linux /proc integration for runtime metric discovery. Elasticsearch transport uses github.com/elastic/elastic-transport-go/v8 as a shared HTTP transport and retry layer. Kafka support pulls in github.com/pierrec/lz4/v4 and github.com/klauspost/compress to enable common broker-side compression codecs for higher-throughput alert streams.

Operational logging and instrumentation dependencies include github.com/go-logr/logr and github.com/go-logr/stdr as structured logging interfaces used by upstream observability libraries, while go.uber.org/multierr complements zap by representing multi-error aggregates. The command-line stack uses github.com/spf13/pflag and github.com/inconshreveable/mousetrap as supporting utilities for flag parsing and Windows console behavior in transitive CLI code paths. The testing ecosystem includes github.com/stretchr/testify for expressive assertions and test helpers used by the project’s verification suite.

Finally, OpenTelemetry transitive packages, including go.opentelemetry.io/otel, go.opentelemetry.io/otel/metric, go.opentelemetry.io/otel/sdk, and go.opentelemetry.io/otel/trace, exist to satisfy upstream client observability hooks and optional tracing/metric interfaces. The dependency google.golang.org/protobuf provides protocol buffer runtime support required by some metrics and transport libraries. Together, these indirect dependencies represent standard, widely used building blocks that improve performance, interoperability, and maintainability without changing the project’s core detection model.
